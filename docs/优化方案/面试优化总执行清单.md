## 0️⃣ 先回答你的问题：Docker / Sentinel / Seata 在单体项目能用吗？

| 技术 | 单体架构能用？ | 说明 |
|------|:---:|------|
| **Docker容器化** | ✅ 完全适用 | Docker跟微服务无关，就是打包运行环境。单体项目容器化是标配。 |
| **Sentinel熔断降级** | ✅ 完全适用 | 熔断保护的是**外部依赖**（AI API、短信API、支付宝API），无论你是不是微服务，调外部服务就需要熔断。 |
| **Seata TCC分布式事务** | ❌ 不适合 | 分布式事务解决的是**跨服务/跨数据库**的一致性问题。单体项目中同一个数据库用 `@Transactional` 就够了。硬加Seata反而是过度设计，面试官会质疑。 |

### 🔄 Seata的替代方案：本地事务 + 补偿 + 幂等（更合适、更真实）

在单体项目中，真正的"分布式"部分只有**支付宝回调**这一个外部依赖。解决方案不是Seata，而是：

```
@Transactional 保证DB操作原子性
    +
支付宝回调幂等处理（防重复通知）
    +
异常补偿定时任务（对账兜底，极端情况用）
```

这个方案反而更能体现你的**架构判断力**——知道什么时候该用什么方案，而不是无脑上分布式事务。

**面试话术**：
> "支付链路我没有盲目上分布式事务。当前是单体架构+单数据库，`@Transactional` 已经覆盖了库存、积分、订单之间的ACID。唯一的外部依赖是支付宝回调——我通过**回调幂等**（业务单号唯一索引）和**定时对账补偿**处理极端情况。分布式事务（Seata TCC）我会在服务拆分、跨库写入时才引入——过早优化是万恶之源。"

## ⚠️ 关于Seata的最终建议

**放弃Seata TCC，改用"本地事务 + 幂等 + 补偿"方案。**

理由：
1. 单体单库场景下 `@Transactional` 已经满足ACID需求
2. Seata需要额外部署TC Server，增加运维复杂度
3. 面试官大概率会追问"为什么单体项目用分布式事务？"——你答不上来反而扣分
4. "本地事务补偿 + 幂等" 体现的是 **架构判断力**，比无脑堆技术更有含金量

**替代实现**：

```java
@Transactional
public void handleAlipayCallback(AlipayNotifyDTO notify) {
    // 1. 幂等：根据支付宝交易号查重
    if (orderService.existsByAlipayTradeNo(notify.getTradeNo())) {
        log.warn("重复回调，忽略: tradeNo={}", notify.getTradeNo());
        return;
    }
    // 2. 更新订单状态
    orderService.markPaid(notify.getOutTradeNo(), notify.getTradeNo());
}
```

极端情况下支付宝回调丢失怎么办？加一个定时任务兜底：

```java
@Scheduled(cron = "0 */10 * * * ?") // 每10分钟
public void reconcileUnpaidOrders() {
    // 查询超过30分钟仍UNPAID的订单
    List<Order> unpaidOrders = orderService.findUnpaidOlderThan(30);
    for (Order order : unpaidOrders) {
        // 主动查支付宝确认支付状态
        boolean paid = alipayService.queryOrderStatus(order.getOutTradeNo());
        if (paid) {
            orderService.markPaidManual(order.getId());
        } else {
            orderService.cancelAndRestoreStock(order.getId());
        }
    }
}
```

**面试话术**：
> "支付一致性我没用Seata。单体单库场景下 `@Transactional` 覆盖了库存、积分、订单的原子操作。支付宝这层我做了**回调幂等**（交易号去重）+ **定时对账补偿**（每10分钟扫超时订单主动查支付宝状态）。这是'最终一致性'的思路——宁可多一次对账，也不加分布式事务的复杂度。等到真的拆微服务、跨库写入了，我才会引入Seata TCC。"
