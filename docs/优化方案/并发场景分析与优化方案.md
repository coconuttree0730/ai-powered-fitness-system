## 二、逐场景深度分析

### 场景1：课程预约 — 已有Redis预留，可升级为"防超卖"亮点 ⭐

**现状**：[BookingServiceImpl](file:///g:/ai-powered-fitness-system-v2/src/main/java/com/fitness/modules/booking/service/impl/BookingServiceImpl.java) 第67行调用 `bookingReservationService.tryReserve()` 做Redis预占位，然后 `persistBooking` 里执行 `sessionMapper.updateBookedCount(sessionId, 1)`。

**当前并发问题**：
```java
// persistBooking 第274行 — 这行没有并发保护！
int updated = sessionMapper.updateBookedCount(dto.getSessionId(), 1);
if (updated == 0) {
    throw new BusinessException(ErrorCode.COURSE_FULL);
}
```

`updateBookedCount` 大概率是 `UPDATE ... SET booked_count = booked_count + 1`，这在DB层面是原子的，但是如果并发请求同时读到 `booked_count = 19, capacity = 20`，两个线程都认为还有一个名额，两个都能通过检查再执行+1，最终 `booked_count = 21 > capacity = 20` → **超卖**。

**你的优势**：Redis预留层已经解决了大部分并发问题（Redis单线程），但Redis故障时降级到纯DB路径存在风险。

**优化方案：在DB层加兜底**

```java
// 在 CourseSessionMapper.xml 中加条件
<update id="updateBookedCount">
    UPDATE course_session
    SET booked_count = booked_count + #{delta}
    WHERE id = #{sessionId}
      AND booked_count + #{delta} <= capacity   <!-- 关键：DB层兜底 -->
      AND capacity > 0
</update>
```

这行SQL改动让 `updateBookedCount` 失败时返回0行，从而触发"名额已满"异常，**DB层面保证绝对不会超卖**。

**面试可说**：
> "课程预约我做了**双层防护**。Redis层用Lua脚本做预占位，能扛住大部分并发流量。DB层 `UPDATE ... WHERE booked_count + delta <= capacity` 做最终兜底——哪怕Redis挂了走降级路径，也不可能超卖。这跟电商秒杀的库存扣减思路一样。"

### 场景2：商品购买 — 🔴 最大并发漏洞，必须修复

**现状**：[ProductOrderServiceImpl.createOrder](file:///g:/ai-powered-fitness-system-v2/src/main/java/com/fitness/modules/product/service/impl/ProductOrderServiceImpl.java) 第103-166行。

**并发问题**：
```java
// 第115行：读取库存
if (product.getStock() < dto.getQuantity()) {
    throw new BusinessException(ErrorCode.PRODUCT_STOCK_INSUFFICIENT);
}

// ... 中间还有价格计算、积分检查等逻辑 ...

// 第137行：扣库存（decreaseStock）
int affected = productMapper.decreaseStock(dto.getProductId(), dto.getQuantity());
```

**核心问题**：第115行读库存 → 中间隔了几十行代码 → 第137行才扣库存。在两个线程同时执行时，都读到 stock=1，都通过检查，都执行扣减 → **超卖**。

**优化方案：DB层原子扣减**

```xml
<!-- ProductMapper.xml -->
<update id="decreaseStock">
    UPDATE product
    SET stock = stock - #{quantity}
    WHERE id = #{productId}
      AND stock >= #{quantity}      <!-- 原子条件 -->
      AND status = 'ACTIVE'
</update>
```

然后去掉前面的 `if (product.getStock() < dto.getQuantity())` 检查（或者保留做快速失败，但要接受它只是"提示"不是"保证"）。

**为什么不用分布式锁？** — 扣库存是高频操作，锁的开销太大。数据库 `WHERE stock >= quantity` 利用行锁天然保证原子性，这是最优解。

**面试可说**：
> "扣库存我只用了一行SQL搞定：`UPDATE product SET stock = stock - ? WHERE id = ? AND stock >= ?`。InnoDB行锁保证同一行同一时刻只有一个事务在修改，`stock >= ?` 条件保证不会扣成负数。很多人上来就加分布式锁，其实数据库行锁+条件UPDATE就够了，更轻量。"

### 场景3：余额支付 — 🔴 并发漏洞，造成资损

**现状**：[MembershipOrderServiceImpl.handleBalancePay](file:///g:/ai-powered-fitness-system-v2/src/main/java/com/fitness/modules/membership/service/impl/MembershipOrderServiceImpl.java) 第289-328行。

**并发问题**：
```java
// 第296行：读余额
BigDecimal balance = user.getBalance();
// 第297行：检查
if (balance.compareTo(order.getPayAmount()) < 0) { throw...; }
// 第302行：设新余额
user.setBalance(balance.subtract(order.getPayAmount()));
// 第303行：写回去
userMapper.updateById(user);
```

两个线程同时扣同一个用户的余额：都读到 balance=100，都通过检查，都扣50后 setBalance=50 → 用户余额应为0但实际是50 → **资损bug**。

**优化方案：CAS更新**

```xml
<!-- UserMapper.xml -->
<update id="deductBalance">
    UPDATE sys_user
    SET balance = balance - #{amount}
    WHERE id = #{userId}
      AND balance >= #{amount}    <!-- CAS条件 -->
</update>
```

```java
// 业务代码中
int affected = userMapper.deductBalance(userId, amount);
if (affected == 0) {
    throw new BusinessException(ErrorCode.INSUFFICIENT_BALANCE);
}
```

**面试可说**：
> "余额扣减的核心是CAS（Compare And Swap）。SQL里 `WHERE balance >= #{amount}` 就是这个CAS条件。如果两个支付请求同时扣一个用户的余额，只有一个能成功（InnoDB行锁互斥），另一个affected=0直接报余额不足。这就是乐观锁在数据库层的实践。"

### 场景5：会员卡激活 — 🟡 防重复激活

**现状**：[MembershipOrderServiceImpl.handleBalancePay](file:///g:/ai-powered-fitness-system-v2/src/main/java/com/fitness/modules/membership/service/impl/MembershipOrderServiceImpl.java) 第325行调用 `userMembershipService.activateMembership(order)`。

如果支付宝回调重复到达（网络重试），或者余额支付的订单被重复调用，同一位用户可能被激活多次会员。

**优化方案：数据库唯一索引兜底**

```sql
-- 一个用户只能有一条活跃的会员记录
CREATE UNIQUE INDEX uk_user_membership_active
ON user_membership(user_id, status)
WHERE status = 'ACTIVE';
```

配合代码：

```java
public void activateMembership(MembershipOrder order) {
    try {
        userMembershipMapper.insert(membership);
    } catch (DuplicateKeyException e) {
        log.warn("会员已激活，跳过重复: userId={}", order.getUserId());
        return; // 幂等处理
    }
}
```

### 🏆 新增场景A：限量优惠券秒杀

```
优惠券：100张，1000人同时抢
```

**实现方案**：

```java
// 预热：活动开始前把库存加载到Redis
redisTemplate.opsForValue().set("coupon:stock:123", "100");

// 扣库存：Lua脚本原子化
String LUA_COUPON_GRAB = """
    local key = KEYS[1]
    local userId = ARGV[1]
    local stock = tonumber(redis.call('GET', key) or '0')
    if stock <= 0 then
        return -1  -- 已抢光
    end
    local already = redis.call('SISMEMBER', key .. ':users', userId)
    if already == 1 then
        return -2  -- 已抢过
    end
    redis.call('DECR', key)
    redis.call('SADD', key .. ':users', userId)
    return redis.call('GET', key)  -- 返回剩余库存
    """;

public Long grabCoupon(Long couponId, Long userId) {
    Long result = redisTemplate.execute(
        new DefaultRedisScript<>(LUA_COUPON_GRAB, Long.class),
        List.of("coupon:stock:" + couponId),
        String.valueOf(userId)
    );
    if (result == -1) throw new BusinessException("已抢光");
    if (result == -2) throw new BusinessException("已抢过");
    // MQ异步写DB持久化
    rocketMQTemplate.asyncSend("coupon-grab-topic", new CouponGrabEvent(couponId, userId));
    return result;
}
```

**面试可说**：
> "秒杀的核心是削峰。我用Redis预扣库存（Lua脚本保证原子），抢到后扔MQ异步持久化到DB。Redis扛住了1000 QPS的瞬时流量，DB只收到100条写入。这是典型的'读写分离'思想——读流量走缓存，写流量走队列。"

### 🏆 新增场景B：热搜词实时Top-N（你已有排行榜基础）

你已经有 [RedisRankingServiceImpl](file:///g:/ai-powered-fitness-system-v2/src/main/java/com/fitness/modules/ranking/service/impl/RedisRankingServiceImpl.java)，可以在它基础上加一个"实时热门课程/商品"功能：

```java
// 用户每次查看课程详情，异步记录热度
@Async
public void recordView(Long courseId) {
    redisTemplate.opsForZSet().incrementScore("hot:courses:daily", 
        "course:" + courseId, 1);
}

// 定时任务：每小时计算热度衰减（让旧数据自然过时）
@Scheduled(cron = "0 0 * * * ?")
public void decayHotScores() {
    // 所有热度分数乘以0.9（指数衰减）
    String LUA = """
        local key = KEYS[1]
        local members = redis.call('ZRANGE', key, 0, -1, 'WITHSCORES')
        for i = 1, #members, 2 do
            local member = members[i]
            local score = tonumber(members[i+1]) * 0.9
            redis.call('ZADD', key, score, member)
        end
        -- 清理低于1分的
        redis.call('ZREMRANGEBYSCORE', key, '-inf', 1)
        return redis.call('ZCARD', key)
        """;
}
```

**面试可说**：
> "实时热榜我用了Redis ZSet + 指数衰减。每次查看课程 `ZINCRBY` 原子加分。每小时跑一次衰减任务——所有分数 ×0.9，低于1分的自动清理。这样做的好处是：不用存储时间窗口数据，只用当前分数就能反映'最近几小时'的热度趋势，存储开销极小。"

### 🏆 新增场景C：库存预热 + 分段库存

如果单个热门商品有1000件库存，10000人同时抢：

```java
// 预热：把库存拆成10个分段，每段100件
for (int i = 0; i < 10; i++) {
    redisTemplate.opsForValue().set("stock:product:1:seg:" + i, "100");
}

// 扣库存：随机选一个分段（减少热点竞争）
String LUA_DEDUCT = """
    local segCount = tonumber(ARGV[1]) or 10
    local qty = tonumber(ARGV[2]) or 1
    
    -- 随机选段
    for i = 1, 10 do
        local seg = math.random(0, segCount - 1)
        local key = KEYS[1] .. ':seg:' .. seg
        local stock = tonumber(redis.call('GET', key) or '0')
        if stock >= qty then
            redis.call('DECRBY', key, qty)
            return seg  -- 返回命中的段号
        end
    end
    
    return -1  -- 全部售罄
    """;
```

**面试可说**：
> "分段库存是解决Redis热Key问题的经典方案。10000个请求如果都打同一个 `stock:1` key，Redis单线程就会成为瓶颈。拆成10个分段后，负载分散到10个key上，竞争强度降到原来的1/10。这跟电商大促的库存分桶原理一样。"

## 五、面试如何回答"你的项目怎么处理并发的"

### 标准回答模板（30秒版）

> "我的项目有多个高并发场景。最重要的两个：一是**课程预约**，热门课放出来几秒就被抢完——我用Redis做双层防护，先Lua脚本预占位减轻DB压力，DB层 `UPDATE WHERE booked_count+1 <= capacity` 做最终兜底。二是**商品/会员购买**——库存和余额扣减我用数据库行锁+CAS条件更新实现原子操作，`UPDATE SET stock=stock-? WHERE id=? AND stock>=?`，一行SQL解决超卖，比分布式锁更轻量。另外我还做了Redis ZSet的实时排行榜和Lua脚本的滑动窗口限流。"

### 被追问"为什么不用分布式锁"

> "分布式锁适合跨服务、跨实例的互斥场景。扣库存、扣余额这种只涉及单表单行的操作，InnoDB行锁天然就是并发的——同一行同时只有一个事务在改。我加 `WHERE stock >= ?` 条件就是利用了行锁机制。分布式锁的代价是额外的网络IO和锁竞争，还有死锁、续期等问题。只有在数据一致性跨多张表时我才用Redisson锁。"

### 被追问"Redis预占和DB扣减的数据一致性怎么保证"

> "这是典型的缓存与数据库一致性问题。我的策略是：Redis预占成功→写DB→如果写DB成功才确认预约。如果Redis预占了但DB失败（比如唯一索引冲突），我会调用 `releaseReservation` 把Redis名额还回去。这里有极小概率Redis还名额的动作也失败，所以我给Redis名额设了TTL，超时自动过期回收。这是**最终一致性**的思路，宁可多还一个名额也不长期占用。"
