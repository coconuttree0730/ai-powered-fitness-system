# 3 系统分析

## 3.1 技术可行性分析

本项目定位为面向健身房场景的智能化管理平台，采用前后端分离架构，后端基于 Spring Boot 3.5.14 构建 RESTful API，前端基于 Vue 3 与 Vite 构建单页应用，并结合 PostgreSQL、Redis、MinIO、Spring AI Alibaba、支付宝沙箱支付等技术组件完成业务闭环。从现有代码结构和依赖配置来看，项目所选技术均属于成熟且社区活跃的主流方案，能够较好支撑健身房运营管理、用户服务和 AI 增值能力的实现，因此系统具备较高的技术可行性。

### 3.1.1 架构与开发技术可行性

系统采用典型的分层与模块化设计。后端按 `user`、`course`、`booking`、`equipment`、`membership`、`product`、`knowledge`、`chat`、`dashboard`、`analysis` 等模块拆分，职责边界清晰，便于团队协作和后期维护。Spring Boot 3 结合 Spring Security、MyBatis-Plus、Flyway 的技术组合，既能满足企业级 Web 应用的快速开发需求，也便于实现统一认证授权、数据持久化和数据库版本管理。前端使用 Vue 3、Vue Router、Pinia、Naive UI、Element Plus 与 ECharts，能够支持游客端、会员端、教练端、管理员端等多角色页面组织，同时具备较好的交互表现力和开发效率。因此，在系统架构设计与软件工程实现层面，本项目具备较强的可落地性。

### 3.1.2 数据存储与基础设施可行性

系统核心业务数据采用 PostgreSQL 存储，适用于课程、预约、器材、会员卡、订单、知识文档等结构化数据管理；Redis 用于缓存、会话辅助和部分 AI 对话状态支持，有助于降低数据库访问压力并提升响应效率；MinIO 用于图片、文档等文件对象存储，适合支撑器材图片、课程封面、知识库上传文档等资源管理需求。数据库迁移使用 Flyway 管理，当前项目已存在较完整的迁移脚本链路，说明系统具备较好的数据演进能力。从部署角度看，该方案对服务器环境要求合理，依赖组件清晰，可在本地、测试环境和生产环境中平滑部署，因此基础设施层面是可行的。

### 3.1.3 AI 能力接入可行性

本项目的核心特色之一是引入 AI 能力辅助健身房服务升级。系统通过 Spring AI Alibaba 对接 DashScope 大模型，实现 AI 健身计划生成、AI 文本润色、AI 运营分析等功能；通过 WebFlux 与 SSE 机制实现会员端流式对话助手；通过 `pgvector` 与本地 Embedding 模型构建 RAG 知识库问答能力，实现对场馆信息、课程规则、会员政策等内容的语义检索；通过 Agent 工具调用封装课程查询、教练查询、会员信息查询等能力，增强对话可用性。上述能力在当前仓库中均已有独立模块、控制器、服务层或数据表支持，说明 AI 方案不是停留在概念层，而是具备直接集成条件，因此 AI 技术路线可行。

### 3.1.4 安全与支付可行性

系统安全体系基于 Spring Security + JWT 实现，支持登录、短信验证码、滑块验证、角色访问控制和接口级权限校验，可以满足健身房管理平台对用户身份认证和资源隔离的基本要求。对于会员卡购买等交易场景，系统已集成支付宝支付能力，并提供支付通知回调接口，具备形成完整业务支付链路的条件。虽然生产环境中仍需进一步加强密钥管理、日志审计、限流和异常告警，但从现有实现看，系统已经具备较好的安全基础和支付接入能力，因此在业务落地层面具备现实可行性。

### 3.1.5 技术可行性结论

综合来看，本项目在技术选型、系统架构、AI 接入、数据存储、安全认证和支付集成等方面均采用了成熟方案，并且已经形成较为完整的代码实现与模块划分。系统所需开发环境与运行环境配置清晰，扩展方向明确，具备继续完善并投入实际使用的技术基础，因此本系统在技术上是可行的。

## 3.2 系统需求分析

本系统的建设目标，不仅是完成传统健身房的基础数字化管理，更是通过 AI 技术提升会员服务质量、教练工作效率和管理者运营决策能力。与普通信息管理系统相比，本项目同时覆盖“前台服务 + 中台运营 + 智能辅助”三类需求，既要满足课程、预约、器材、会员卡等日常业务处理，也要满足知识问答、个性化健身计划生成、数据分析报告等智能化应用场景。

从业务视角看，系统需要解决以下几个实际问题：一是游客和会员获取场馆信息、课程信息、教练信息的效率不高；二是人工安排课程、管理预约和维护器材报修的成本较高；三是会员服务缺少个性化与连续性，难以形成长期粘性；四是管理端对于营收、预约、器材状态、用户增长等数据缺乏统一可视化分析工具。基于这些问题，系统必须建立统一的用户体系、业务流程体系和数据支撑体系，使健身房从信息展示、服务受理到智能分析形成完整闭环。

在系统范围上，本项目面向四类主要使用对象展开设计：游客、会员、教练和管理员。游客以信息浏览为主，会员以服务消费和智能辅助为主，教练以课程与学员管理为主，管理员以后台配置和经营分析为主。系统因此需要兼顾公开访问能力与登录后业务处理能力，并通过角色权限控制保证不同用户只能访问各自职责范围内的数据与功能。

## 3.3 功能需求分析

结合当前项目实现情况，系统功能需求可归纳为以下几个方面：

### 3.3.1 用户与认证功能

系统需要提供用户注册、登录、身份校验和角色识别能力。用户可通过普通登录、短信验证码登录等方式进入系统，登录前引入滑块验证机制以降低恶意请求风险。登录成功后，系统应根据用户角色将其引导至对应功能区域，并通过 JWT 令牌维持登录状态。对于会员和教练，还需要支持个人资料维护、头像与基础信息展示、健身档案管理等功能。

### 3.3.2 课程与预约功能

系统需要支持公开课程展示、课程详情查看、课程分类管理、课程创建与修改、课程预约与取消、课程容量控制以及教练排期管理。游客可浏览公开课程信息，会员登录后可预约课程并查看个人预约记录，教练和管理员可维护课程内容与排班信息。对于扩展训练场景，系统还应支持视频课程管理，以满足非到店训练和内容服务的需求。

### 3.3.3 器材与报修功能

系统需要提供器材信息维护、器材详情展示、器材状态管理、故障报修、维修记录跟踪等功能。游客与会员可查看器材基础信息，会员在发现问题后可提交报修申请并查询处理进度，管理员可统一管理器材状态、维修流程和处理结果。该功能有助于减少器材故障长期堆积，提高场馆设备的可用率和服务质量。

### 3.3.4 会员卡、商品与订单功能

系统需要支持会员卡展示、会员卡推荐、会员卡详情查看、会员卡下单与支付、订单查询、会员有效性校验等功能，并支持健身商品浏览、价格计算、商品订单创建、支付与取消等流程。会员卡功能是本项目的重要营收入口，系统应能够记录会员购买结果、会员生效状态和订单流转状态。通过接入支付模块，系统可以形成从商品或会员卡展示到下单支付的完整消费链路。

### 3.3.5 AI 健身服务功能

系统需要根据会员的身体信息、训练目标和经验水平，为其自动生成个性化健身计划，并支持保存与查看历史计划。与此同时，系统应提供 AI 健身助手，通过流式对话方式回答会员关于训练安排、课程选择、器材使用、场馆规则等问题。为了提升回答质量，AI 助手还需要具备知识库检索、工具调用和长期记忆等能力，从而增强连续对话体验和内容准确性。

### 3.3.6 知识库与智能问答功能

系统需要支持知识文档上传、分类管理、发布归档、重新索引、向量分块与语义检索。管理员可将场馆制度、会员政策、课程说明、常见问题等资料维护到知识库中，系统通过 RAG 机制完成知识问答，使 AI 回复能够基于业务资料而不是完全依赖大模型自由生成。这一能力能够提升客服效率，并降低人工重复答疑成本。

### 3.3.7 运营看板与分析功能

系统需要支持后台数据看板展示，包括会员卡销量、用户增长、课程统计、营收趋势、到店高峰时段、器材状态和报修统计等内容。除基础图表外，系统还应支持基于运营数据自动生成 AI 分析报告，并对报告进行保存、查询和删除管理，以便管理者快速把握经营状态并辅助决策。

### 3.3.8 内容与基础配置功能

系统需要支持公告管理、轮播图管理、教练信息维护、数据字典维护等基础内容管理能力。该类功能虽然不直接产生交易，但对首页展示、平台运营、规则说明和统一配置具有基础支撑作用，是系统完整运行不可缺少的组成部分。

## 3.4 非功能需求分析

### 3.4.1 性能需求

系统需要能够支持健身房日常运营中的并发访问需求，尤其是在课程浏览、课程预约、会员卡购买和 AI 对话等场景下，应保持较稳定的响应性能。对于普通查询类接口，应尽可能在较短时间内返回结果；对于 AI 对话和 AI 报告生成等耗时操作，应通过流式响应、异步处理或合理超时控制优化用户体验。Redis 缓存、数据库连接池和分页查询机制应共同承担性能优化任务。

### 3.4.2 安全性需求

系统需要保证用户身份信息、订单信息、会员档案和业务数据的安全。为此，系统应具备登录认证、角色授权、密码加密、Token 校验、接口访问控制等基础安全机制；对于支付回调、文件上传和 AI 输入内容，还需要防止伪造请求、恶意上传和内容注入等风险。由于系统涉及会员消费和个人健身档案，安全性是平台上线的关键前提之一。

### 3.4.3 可靠性需求

系统需要保持稳定运行，避免因单点故障导致关键业务中断。数据库迁移需要可追溯，重要业务数据需要能够持久保存；预约、订单、会员状态、知识文档等核心数据不应因系统异常而轻易丢失。对于 AI 能力和外部支付接口，还应考虑接口超时、第三方服务波动时的降级处理或重试策略，以保证系统核心功能可持续使用。

### 3.4.4 可维护性需求

系统需要便于后续迭代开发和功能扩展。当前项目已采用模块化分层结构，控制器、服务、实体、Mapper、前端页面和 API 调用逻辑划分较清晰，这为后续增加业务模块、调整数据结构和优化界面交互提供了基础。数据库通过 Flyway 管理版本，便于多人协作和环境同步；统一的角色权限设计也有利于后期补充更细粒度的授权规则。

### 3.4.5 可扩展性需求

随着业务发展，系统可能需要继续扩展更多课程类型、更多支付方式、更复杂的会员权益机制以及更强的 AI 场景能力。因此，系统在架构上需要支持模块独立演进，并能够较方便地接入新的第三方服务或模型能力。当前前后端分离、对象存储、向量检索和模块化设计的方案，为后期横向扩展提供了良好基础。

### 3.4.6 易用性需求

系统面向的用户群体包含普通会员、健身教练和后台管理人员，信息技术水平差异较大，因此界面应尽量简洁清晰、操作路径明确。游客端应突出课程、教练、器材等公开信息浏览体验；会员端应突出预约、购卡、计划生成和助手服务；教练端应突出课程和学员管理；管理员端应突出配置、统计和审核。只有在界面交互与业务流程上保持足够直观，系统才能真正提高使用效率。

## 3.5 用户角色分析

### 3.5.1 游客

游客是未登录用户，主要需求是快速了解健身房的基本情况，包括首页展示内容、公开课程、器材信息和教练详情等。该角色不参与核心业务处理，但承担平台引流和转化入口作用，因此系统应为其提供稳定、清晰、易浏览的公开信息访问能力。

### 3.5.2 会员

会员是系统中最核心的服务对象，既是课程预约者、会员卡购买者，也是 AI 健身服务的主要使用者。会员需要完成注册登录、查看个人档案、预约课程、提交报修、购买会员卡或商品、生成个性化健身计划、使用 AI 助手问答等操作。系统在设计时应重点保障会员端的功能完整性、操作便捷性和数据安全性，以提升用户粘性与服务体验。

### 3.5.3 教练

教练是连接平台服务与线下训练的重要角色。其主要需求包括维护个人主页信息、管理课程、查看学员情况、安排课程日程以及管理私教相关服务内容。系统应帮助教练减少重复性的排课与信息维护工作，提高课程组织效率，并使会员能够更方便地了解教练特长和服务内容。

### 3.5.4 管理员

管理员负责整个平台的后台运营与维护，是权限最高的角色。其主要职责包括用户管理、课程管理、器材管理、会员卡管理、订单管理、知识库维护、公告与轮播图维护、数据字典管理以及数据分析报告查看。系统需要为管理员提供较完整的控制台能力，使其能够从经营、内容、设备和服务多个维度统筹平台运行。

### 3.5.5 用户角色分析结论

本系统的用户角色划分与健身房实际业务流程较为匹配，四类角色在权限范围、使用场景和关注重点上具有明显差异。通过角色隔离与页面分区设计，系统能够在同一平台中同时满足公开展示、会员服务、教练协同和后台运营四类需求，从而构建完整的智能健身房数字化服务体系。
