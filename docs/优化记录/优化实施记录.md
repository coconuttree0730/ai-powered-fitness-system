### 1.1 UserMapper.java — 新增 3 个原子操作方法

**文件:** `src/main/java/com/fitness/modules/user/mapper/UserMapper.java`

```java
@Update("UPDATE sys_user SET balance = balance - #{amount}, update_time = NOW() " +
        "WHERE id = #{userId} AND balance >= #{amount}")
int deductBalance(@Param("userId") Long userId, @Param("amount") BigDecimal amount);
```

## 三、数据库索引优化

### 3.1 新建 Flyway 迁移 V50

**文件:** `src/main/resources/db/migration/V50__add_dashboard_composite_indexes.sql`

```sql
-- product_order: 覆盖 countTodayOrders, sumTodayRevenue, selectRevenueTrend
CREATE INDEX idx_product_order_created_at_status
    ON product_order(created_at, status)
    WHERE status IN ('PAID', 'SHIPPED', 'COMPLETED');

    ON fitness_booking(user_id, booking_time)
    WHERE deleted = false;

    ON fitness_booking(course_id, status)
    WHERE deleted = false;
```

### 3.2 修复 DashboardMapper.xml 中的 DATE() 反模式

**文件:** `src/main/resources/mapper/dashboard/DashboardMapper.xml`

将所有 `DATE(created_at) = CURRENT_DATE` 改为 `created_at >= CURRENT_DATE AND created_at < CURRENT_DATE + INTERVAL '1 day'`。

涉及查询：`countTodayOrders`、`sumTodayRevenue`、`selectRevenueTrend`（hour/day 两个分支）、`selectUserGrowth`。

### 面试话术

> "用 EXPLAIN ANALYZE 分析 Dashboard 查询，发现 `DATE(created_at)` 包裹导致索引失效——PostgreSQL 无法对函数计算结果使用 B-tree 索引。改为范围谓词 `>= X AND < Y+1`（sargable），并创建 4 个部分复合索引覆盖 WHERE + SELECT 列。"

## 七、Redis 接口限流

### 7.1 @RateLimit 注解

**文件:** `src/main/java/com/fitness/common/ratelimit/RateLimit.java`（新建）

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RateLimit {
    String key();                              // 限流键前缀
    int limit() default 10;                    // 窗口内最大请求数
    int window() default 60;                   // 窗口大小（秒）
    RateLimitDimension dimension() default RateLimitDimension.IP; // 限流维度
}
```

### 7.2 RateLimitDimension 枚举

**文件:** `src/main/java/com/fitness/common/ratelimit/RateLimitDimension.java`（新建）

支持 `IP`、`USER`、`CUSTOM` 三种维度。`CUSTOM` 支持 SpEL 表达式从方法参数动态解析限流键。

### 7.3 RateLimitAspect — Redis Lua 滑动窗口

**文件:** `src/main/java/com/fitness/common/ratelimit/RateLimitAspect.java`（新建）

核心算法：Redis Lua 脚本实现滑动窗口限流（`ZREMRANGEBYSCORE` → `ZCARD` → `ZADD` → `EXPIRE`），单次 Redis 调用保证原子性。

### 7.4 限流端点

| Controller | 端点 | 限流策略 | 维度 |
|------------|------|----------|------|
| `AuthController` | `POST /auth/register` | 5 次/60s | IP |
| `AuthController` | `POST /auth/login` | 10 次/60s | IP |
| `AuthController` | `POST /auth/sms-code` | 10 次/60s | IP |
| `ChatAssistantController` | `POST /chat/messages` | 20 次/60s | USER |
| `ChatAssistantController` | `POST /chat/messages/stream` | 10 次/60s | USER |
| `ChatAssistantController` | `POST /chat/fitness-plan/generate` | 5 次/60s | USER |

### 面试话术

> "用自定义 @RateLimit 注解 + Spring AOP 切面实现声明式接口限流。底层用 Redis Lua 脚本实现滑动窗口算法——ZREMRANGEBYSCORE 移除过期记录、ZCARD 统计当前窗口请求数、ZADD 添加新请求，整个操作在 Redis 服务端单次执行保证原子性。支持 IP/USER/自定义 SpEL 三种维度，注册接口防刷用 IP 维度，AI 对话用 USER 维度防止单用户占用过多资源。"

## 八、多级缓存 Caffeine(L1) + Redis(L2)

### 8.1 依赖

**文件:** `pom.xml`

新增 `com.github.ben-manes.caffeine:caffeine`（Spring Boot 管理版本）。

### 8.2 RedisTemplateCacheSupport 改造

**文件:** `src/main/java/com/fitness/common/cache/RedisTemplateCacheSupport.java`

在原有 Redis 手动缓存之上增加 Caffeine L1 本地缓存层：

| 配置项 | 值 | 说明 |
|--------|------|------|
| `maximumSize` | 2000 | 最大缓存条目数 |
| `expireAfterWrite` | 180s (3min) | 写入后过期，比 Redis TTL 短 |
| `recordStats` | true | 启用命中率统计 |

**查询流程：**
1. **L1 Caffeine** → 命中直接返回（~纳秒级）
2. **L2 Redis** → 命中回填 L1 后返回
3. **分布式锁** → 防止缓存击穿（double-check + retry）
4. **Loader** → 加载数据同时写入 L1 + L2

**驱逐策略：**
- `evict(key)` → 同时失效 L1 + Redis
- `evictAll(cacheName)` → Redis 按前缀删除 + L1 全量失效（Caffeine 不支持前缀匹配，但 L1 TTL 短，3 分钟内自动过期）

**监控：**
- `getL1Stats()` → 返回 Caffeine CacheStats（hitRate, evictionCount 等）
- `getL1Size()` → 返回 L1 当前条目数

### 面试话术

> "热数据缓存采用 Caffeine L1 + Redis L2 双层架构。L1 是 JVM 堆内缓存，命中延迟 ~纳秒级，TTL 设为 3 分钟（比 Redis 短），用较短的过期时间换取数据一致性。查询路径：L1 → Redis → DB，Redis 命中自动回填 L1。缓存击穿防护用 Redis 分布式锁 + double-check，14 个热点缓存区全部覆盖。evictAll 时 L1 全量失效——因为 Caffeine 不支持前缀匹配，但 3 分钟 TTL 保证数据不会长期脏读。"

## 十一、敏感数据脱敏

### 11.1 方案设计

基于 Jackson 注解的声明式脱敏方案。在 VO 字段上标注 `@Sensitive(type)` 注解，序列化时根据当前用户角色自动决定是否脱敏。零侵入 Service 层，脱敏逻辑与业务代码完全解耦。

### 11.2 核心组件

| 文件 | 说明 |
|------|------|
| `src/main/java/.../common/sensitive/SensitiveType.java` | 敏感类型枚举（PHONE, EMAIL, NAME, ADDRESS, ID_CARD, BANK_CARD, PASSWORD） |
| `src/main/java/.../common/sensitive/MaskUtil.java` | 脱敏工具类，7 种脱敏算法 |
| `src/main/java/.../common/sensitive/Sensitive.java` | `@Sensitive` 注解，`@JacksonAnnotationsInside` 元注解 |
| `src/main/java/.../common/sensitive/SensitiveSerializer.java` | Jackson `ContextualSerializer`，读取 `SecurityContextHolder` 判断角色 |
| `src/main/java/.../common/sensitive/SensitiveModule.java` | Jackson Module，注册到 ObjectMapper |

### 11.3 角色判断逻辑

`SensitiveSerializer.shouldMask()` 检查 `SecurityContextHolder` 中的认证信息：
- `ROLE_ADMIN` / `ROLE_COACH` → 返回原始数据
- 其他角色（MEMBER、未认证）→ 返回脱敏数据

### 11.4 VO 注解标注

| VO | 字段 | 脱敏类型 |
|----|------|----------|
| `UserVO` | phone | PHONE (`138****1234`) |
| `UserVO` | email | EMAIL (`zha***@qq.com`) |
| `CoachDetailVO` | realName | NAME (`张*明`) |
| `CoachDetailVO` | phone | PHONE |
| `CoachDetailVO` | email | EMAIL |
| `CoachHomePageVO` | realName | NAME |
| `CoachHomePageVO` | phone | PHONE |
| `CoachHomePageVO` | email | EMAIL |
| `ProductOrderVO` | address | ADDRESS (`北京市朝阳区***`) |

### 11.5 测试

- `MaskUtilTest.java` — 7 种脱敏算法的单元测试
- `SensitiveSerializerTest.java` — 无认证返回脱敏数据、ADMIN 角色返回原始数据

### 11.6 面试话术

> "敏感数据脱敏用的是 Jackson 自定义序列化器方案。在 VO 字段上标注 `@Sensitive(SensitiveType.PHONE)` 注解，底层是 `ContextualSerializer`，序列化时从 `SecurityContextHolder` 读取当前用户角色——管理员看到完整数据，普通用户看到脱敏数据。整个过程零侵入 Service 层，脱敏逻辑和业务代码完全解耦。相比在 Service 层手动调 `maskPhone()`，声明式方案更易维护，新增字段只需加一行注解。"
